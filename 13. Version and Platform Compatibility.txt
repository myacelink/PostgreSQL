◼︎ 이전 PostgreSQL 버전
array_nulls(부울)
이것은 배열 입력 파서가 null 배열 요소를 지정하는 것으로 인용되지 않은 NULL을 인식하는지 여부를 제어합니다. 기본적으로 이것은 켜져 있으며 null 값이 포함된 배열 값을 입력할 수 있습니다. 그러나 8.2 이전의 PostgreSQL 버전은 배열에서 null 값을 지원하지 않았으므로 NULL을 문자열 값이 "NULL"인 일반 배열 요소를 지정하는 것으로 취급합니다. 이전 동작이 필요한 응용 프로그램과의 이전 버전과의 호환성을 위해 이 변수를 끌 수 있습니다.

이 변수가 꺼져 있는 경우에도 null 값을 포함하는 배열 값을 생성할 수 있습니다.

backslash_quote(enum)
이것은 문자열 리터럴에서 따옴표를 \'로 나타낼 수 있는지 여부를 제어합니다. 따옴표를 표시하는 데 선호되는 SQL 표준 방법은 따옴표를 두 배로 만드는 것('')이지만 PostgreSQL은 역사적으로 \'도 허용했습니다. 그러나 일부 클라이언트 문자 집합 인코딩에는 마지막 바이트가 ASCII \와 수치적으로 동일한 멀티바이트 문자가 있기 때문에 \'를 사용하면 보안 위험이 발생합니다. 클라이언트 측 코드가 잘못 이스케이프되면 SQL 주입 공격이 가능합니다. 이 위험은 인용 부호가 백슬래시로 이스케이프된 것처럼 보이는 쿼리를 서버에서 거부하도록 함으로써 방지할 수 있습니다. backslash_quote의 허용되는 값은 on(항상 \' 허용), off(항상 거부) 및 safe_encoding(클라이언트 인코딩이 멀티바이트 문자 내에서 ASCII \를 허용하지 않는 경우에만 허용)입니다. safe_encoding은 기본 설정입니다.

참고
표준 준수 문자열 리터럴에서 \는 어쨌든 \를 의미합니다. 이 매개변수는 이스케이프 문자열 구문(E'...')을 포함하여 비표준 준수 리터럴 처리에만 영향을 줍니다.

escape_string_warning(boolean)
켜져 있으면 백슬래시(\)가 일반 문자열 리터럴('...' 구문)에 나타나고 standard_conforming_strings가 꺼져 있으면 경고가 발행됩니다. 기본값은 켜짐입니다.

백슬래시를 이스케이프로 사용하려는 응용 프로그램은 이스케이프 문자열 구문(E'...')을 사용하도록 수정해야 합니다. 일반 문자열의 기본 동작은 이제 SQL 표준에 따라 백슬래시를 일반 문자로 취급하기 때문입니다. 이 변수는 변경해야 하는 코드를 찾는 데 도움이 되도록 활성화할 수 있습니다.

lo_compat_privileges(boolean)
PostgreSQL 9.0 이전 릴리스에서는 대형 객체에 액세스 권한이 없었으므로 모든 사용자가 항상 읽고 쓸 수 있었습니다. 이 변수를 on으로 설정하면 이전 릴리스와의 호환성을 위해 새 권한 검사가 비활성화됩니다. 기본값은 꺼져 있습니다. 수퍼유저만 이 설정을 변경할 수 있습니다.

이 변수를 설정해도 큰 개체와 관련된 모든 보안 검사가 비활성화되지 않으며 PostgreSQL 9.0에서 기본 동작이 변경된 경우에만 비활성화됩니다.

operator_precedence_warning(boolean)
켜져 있으면 구문 분석기는 연산자 우선 순위 변경의 결과로 PostgreSQL 9.4 이후 의미가 변경되었을 수 있는 모든 구성에 대해 경고를 내보냅니다. 이는 우선 순위 변경으로 인해 손상된 것이 있는지 확인하기 위해 응용 프로그램을 감사하는 데 유용합니다. 그러나 완벽하게 유효하고 표준을 준수하는 SQL 코드에 대해 경고하므로 프로덕션 환경에서 계속 켜져 있는 것은 아닙니다. 기본값은 꺼져 있습니다.

See Section 4.1.6 for more information.
Section 4.1.6 : https://www.postgresql.org/docs/12/sql-syntax-lexical.html#SQL-PRECEDENCE
자세한 내용은 섹션 4.1.6을 참조하십시오.
섹션 4.1.6 : https://www.postgresql.org/docs/12/sql-syntax-lexical.html#SQL-PRECEDENCE

quote_all_identifiers(boolean)
데이터베이스가 SQL을 생성할 때 (현재) 키워드가 아니더라도 모든 식별자를 따옴표로 묶습니다. 이것은 EXPLAIN의 출력뿐만 아니라 pg_get_viewdef와 같은 함수의 결과에도 영향을 미칩니다. pg_dump 및 pg_dumpall의 --quote-all-identifiers 옵션도 참조하십시오.

standard_conforming_strings(boolean)
이것은 일반 문자열 리터럴('...')이 SQL 표준에 지정된 대로 백슬래시를 문자 그대로 처리하는지 여부를 제어합니다. PostgreSQL 9.1부터 기본값은 켜짐(이전 릴리스의 기본값은 꺼짐)입니다. 애플리케이션은 이 매개변수를 확인하여 문자열 리터럴이 처리되는 방법을 결정할 수 있습니다. 이 매개변수의 존재는 이스케이프 문자열 구문(E'...')이 지원된다는 표시로 간주될 수도 있습니다. 이스케이프 문자열 구문(섹션 4.1.2.2)은 애플리케이션에서 백슬래시가 이스케이프 문자로 처리되기를 원하는 경우 사용해야 합니다.
섹션 4.1.2.2 : https://www.postgresql.org/docs/12/sql-syntax-lexical.html#SQL-SYNTAX-STRINGS-ESCAPE

synchronize_seqscans(boolean)
이를 통해 대형 테이블의 순차 스캔이 서로 동기화되어 동시 스캔이 거의 동시에 동일한 블록을 읽고 따라서 I/O 워크로드를 공유할 수 있습니다. 이것이 활성화되면 스캔은 테이블 중간에서 시작된 다음 이미 진행 중인 스캔 활동과 동기화하기 위해 모든 행을 포함하도록 끝을 "감싸기"할 수 있습니다. 이로 인해 ORDER BY 절이 없는 쿼리에서 반환된 행 순서가 예기치 않게 변경될 수 있습니다. 이 매개변수를 off로 설정하면 순차 스캔이 항상 테이블의 시작 부분에서 시작하는 pre-8.3 동작이 보장됩니다. 기본값은 켜짐입니다.

◼︎ 플랫폼 및 클라이언트 호환성
transform_null_equals(부울)
설정하면 expr = NULL(또는 NULL = expr) 형식의 식이 expr IS NULL로 처리됩니다. 즉, expr이 null 값으로 평가되면 true를 반환하고 그렇지 않으면 false를 반환합니다. expr = NULL의 올바른 SQL 사양 호환 동작은 항상 null(알 수 없음)을 반환하는 것입니다. 따라서 이 매개변수는 기본적으로 꺼짐으로 설정됩니다.

그러나 Microsoft Access에서 필터링된 양식은 null 값을 테스트하기 위해 expr = NULL을 사용하는 것으로 보이는 쿼리를 생성하므로 해당 인터페이스를 사용하여 데이터베이스에 액세스하는 경우 이 옵션을 켜야 할 수 있습니다. expr = NULL 형식의 식은 항상 null 값(SQL 표준 해석 사용)을 반환하기 때문에 그다지 유용하지 않고 일반 응용 프로그램에서 자주 나타나지 않으므로 이 옵션은 실제로 거의 해를 끼치지 않습니다. 그러나 새 사용자는 null 값과 관련된 표현식의 의미에 대해 자주 혼동하므로 이 옵션은 기본적으로 꺼져 있습니다.

이 옵션은 정확한 형식 = NULL에만 영향을 미치며, 다른 비교 연산자 또는 등호 연산자(예: IN)를 포함하는 일부 표현식과 계산적으로 동일한 기타 표현식에는 영향을 미치지 않습니다. 따라서 이 옵션은 잘못된 프로그래밍에 대한 일반적인 수정 사항이 아닙니다.

관련 정보는 섹션 9.2를 참조하십시오.
